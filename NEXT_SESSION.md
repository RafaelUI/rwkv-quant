# Continuation prompt — rwkv-quant

Квантование RWKV-7 под Apple Silicon (M4 base, 16GB, БЕЗВЕНТИЛЯТОРНЫЙ Air).
Цель: максимальная производительность decode при минимальной потере качества.
Формат `.rwkvq`, бэкенд Metal (MLX), референс-модель
`~/Develop/WKV-kvant/rwkv7-g1h-1.5b-ctx10240.pth` (1.5B, world naming, ~2.8GB).
Полный лог сессий до 19.07 включительно — в git-истории этого файла.

## Законы методологии (нарушались, приводило к ложным выводам)

1. Все замеры скорости — ТОЛЬКО A/B-чередованием в одном процессе.
   Безвентиляторный дрейф под нагрузкой до 1.8x на «том же» замере
   (tmix N=1: 3.5–6.3 мс между процессами, 19.07-15).
2. Тяжёлые ppl-прогоны 1.5B — по одному конфигу на процесс
   (`tests/diagnose_one.py <case>`); не тащить лишние модели в процесс
   (16GB, bf16-чекпоинт 2.8GB; своп-инциденты были).
3. ppl сравнивать только внутри текущего корпуса
   (eval_corpus_world.pt[:8] из ~/Develop/test.txt). Числа других эпох —
   перемерять.
4. В циклах с деквантом — eval каждую итерацию.
5. Комбинации конфигов непредсказуемы: изолированный выигрыш не
   переносится в композит без замера. Отрицательные результаты не
   удалять (nb2-кернель, qs/qm-упаковка, pipeline decode — в коде/логах).
6. Metal System Trace: «Frame» в GUI-иерархии != command buffer.
   Точные числа — через `xcrun xctrace export --xpath '...schema=
   "metal-application-command-buffer-submissions"'` + union интервалов
   (не sum), с учётом id/ref-интернинга XML.
7. У eval_corpus_world.pt строки по 128 токенов: `data[:, :1024]` МОЛЧА
   даёт T=128. Любое «слишком хорошее» число сверять с руфлайном ДО
   радости (3.1 TFLOP/prefill-проход @ потолок 2733 GFLOPS = максимум
   ~910 tok/s на 1.5B; полоса 120 GB/s = пол decode ~7 мс/ток на 843MB).
8. mx-массивы иммутабельны: снапшот state = хранение ссылки, откат
   бесплатен (используется в спекулятивке).

## Разгадка «Molly быстрее при большем размере» (19.07-11, закрыто)

1. Кернель: mx.quantized_matmul на рабочих формах (OUT=2048/8192 — 144
   GEMV/ток) быстрее нашего на 37–42% per-byte (izolated A/B,
   bench_mlx_vs_gw_ab.py: tmix 62.7 vs 89.0 GB/s indep). На head —
   паритет (95 vs 100 GB/s, оба у практического потолка ~80–85% DRAM;
   больше не выжимает никто — llama.cpp та же лига).
2. Префилл: у MLX деквант фьюзнут в GEMM; наш GEMM-путь платит деквант
   отдельным проходом (23–35% времени): 2087/1683 GFLOPS vs 2638/2511.
3. Это НЕ launch-щели и не GPU idle (98.99% занятости, закрыто 19.07-3).

Качество (бит-экзакт mx.quantize на G1H, eval_mlx_affine_real.py):
affine 6-бит = +1.32% (vs +0.12% у REDUCTION v2 при том же размере);
affine p5c4e5 (размер COMPRESSION) = +10.1% (vs +2.47%). «Фантомный
ориентир +0.07%» был льстивой эмуляцией (fp32-скейлы). ВЫВОД: sb6+AW
выигрывает Парето качество/размер, MLX выигрывает рантайм; смена
формата ради скорости качеством не окупается. Скорость добывать
кернелем при родном формате.

## N-батчевый режим gw-кернеля (19.07-14/15, РЕАЛИЗОВАН)

quant_linear_gw.py: `_get_kernel_gw_nb` — NN колонок на threadgroup,
веса декодируются один раз. Дисптчер: N=1 старый кернель (НЕ ТРОНУТ);
2<=N<128 — чанки по NB_CHUNK=4, RB по форме (`_rb_nb`: IN>=8192 → rb8
[value], OUT>=8192 → rb4 [key], иначе rb2); N>=128 — dequant-GEMM
(GEMM_MIN_BATCH_NB; старый обрыв «полный деквант с N=16» устранён).
nb2-вариант (подъём загрузок, декод ALU-дублируется) — ХУЖЕ, за флагом
NB_V2=False как отрицательный результат.

ГЕЙТЫ: tests/test_gw_nb_parity.py — БИТ-В-БИТ со старым поколоночным
(int4/int5/int6 x N=2..12, включая head и int6). Полный регресс
зелёный, decode N=1 не задет (15.5 FUSE / 16.4 compiled).

Кернельная цена колонки при N=4 (bench_gw_nb_sweep/groups/cmix_split):
tmix 2.3 | cmix key 2.26 (rb4) | cmix value 2.85 (rb8) | head 0.68
против 5.3 / 3.1 / 3.1 / 1.3 при N=1. До MLX qmv (T=4 = 1.2x) не
дотянули: чанк=4 упирается в register spill, дальше — кернель-3.

Кривая верификации чемпиона (T=k, eager, bench_verify_cost.py):
T=2 1.16x | T=4 1.99x | T=8 3.41x | T=16 123 мс | T=32 235 мс;
полка ~7.3–7.9 мс/ток при T>=12 (bound: WKV/LoRA/обвязка + кернель 2x
от идеала). Compiled step срезает ~3 мс с прохода (кэш по формам).

## Спекулятивный декодинг (19.07-12/13/15, PoC работает)

Механика (spec_decode_poc/champ/draft2.py), greedy, БЕЗ правок модели:
pending-трюк (state обновляется только при полном принятии, принятое
догоняет state следующим проходом), у драфта снапшот-откат (закон 8),
верификация compiled step + tail_only (head только на k+1 хвостовых,
бит-в-бит, quant_model.py). Выход самосогласован; траектория T=k может
расходиться с T=1 на near-tie логитах (fp16 WKV-чанк) — «совпадение с
plain» НЕ гейт; гейт — бит-точность линейных слоёв.

| вариант | мс/ток | speedup | приёмка |
|---|---|---|---|
| ЧЕМПИОН + n-gram (без драфт-модели), k=4 | 15.27 | 1.09x | 87% (цикл. текст) |
| affine6 + n-gram, k=4 | 13.34 | 1.25x | 81% (накачано циклами) |
| affine6 + драфт g1d-0.1b, k=2 | 14.91 | 1.12x | 72% честная (48–88%) |

Драфт: `~/Develop/WKV-kvant/rwkv7-g1d-0.1b-ctx8192.pth` (world, L12-D768,
грузится штатным QuantRWKV7 c n_layer=12/n_embd=768, bf16-dense).
Драфт-шаг 4.61 мс compiled = launch-floor рантайма (по полосе ~2 мс) —
на текущих floor'ах модельный драфт в нуле, n-gram (бесплатный) в плюсе
только на повторах. ru60m НЕ драфт (словарь ru16k != world).
Спекулятивка = мультипликатор на будущий кернель/рантайм-фикс, не
альтернатива; полка кривой ограничивает её ~2.1x сверху.

## Закрытые темы (не переоткрывать без новых данных)

- ANE/CoreML: нет выигрыша на нашем профиле + toolchain нежизнеспособен
  на 16GB (OOM при конвертации даже 0.4B). Только прямой доступ
  (Orion/ANEForge) и на другой машине.
- GPU idle/командная граница: занятость 98.99%, резерва нет.
- qs/qm 6-бит в кернеле: медленнее (instruction count > экономия байт).
- Пайплайн decode (eval раз в 8/16): MLX уже перекрывает хост-синк.
- 4x-сжатие (<=738MB) при <=5% ppl: недостижимо в нибл-формате
  (пол ~887MB = +5.62%). Нужна суб-ниббл/sparse-техника (не начата).
- MXFP4-семантика: закрыта по качеству.
- v3 simdgroup ОХОТА ЗА ПОЛОСОЙ на head: паритет, head оптимален.
- nb2 (подъём загрузок): хуже nb1, за флагом.
- GEMM r/k/v-фьюз префилла: выигрыш ~2–3%, решение А. — не делать;
  префилл чемпиона near-optimal (99% потолка матмула в cmix).
- Гибрид «тело в mlx affine ради скорости»: качество не окупает
  (см. разгадку Molly выше).

## Открытые вопросы / следующие шаги (приоритет по цене/выгоде)

1. КЕРНЕЛЬ-3: раскладка MLX qmv (8 строк/TG, квад-редукция, simdgroup;
   исходник-образец: mlx/backend/metal/kernels/quantized.h, PR #1503
   «Fast Small QMV»). Прогноз: verify T=4 1.99x → ~1.2–1.4x, spec
   1.09x → ~1.3–1.5x (=> ~11–12 мс/ток COMPRESSION с родным качеством),
   возможно +GB/s и в N=1 (62.7→~85 на средних формах = −2..3 мс decode).
   Самая большая работа, самый большой приз.
2. Баг REDUCTION-пресета старого образца был починен переездом на v2
   (presets.py уже = proj6-noAW/emb6-AW/cmix6-AW), но докстринги/CLI
   стоит перепроверить перед релизом; eval_v2_real/eval_v2_kernel
   оформить из прототипов.
3. nb-режим для quant_linear_v2 (g_lora/small int8) — та же хирургия,
   что 19.07-14, выгода ~1–2 мс на verify-раунд.
4. Драфт-спекулятивка: вернуться ПОСЛЕ кернеля-3/снижения floor'ов;
   для temp>0 нужен rejection sampling. n-gram (prompt-lookup) можно
   включать в прод-генерацию уже сейчас: 1.1–1.25x на повторах,
   ~1.0x без них, проигрыша нет.
5. AW-автоправило по битности/группе (asym_sb6 vs _aw) вместо ручного
   оверрайда в каждом QuantConfig (proj@6 без AW, emb/cmix с AW).
6. Опц.: canonical INT6 на G1 bf16 для чистого G1-vs-G1 в отчёте.
7. Гетерогенный GPU+CPU/ANE параллелизм: свежая литература говорит
   «нет бесплатной параллельной полосы DRAM» (M4 Pro, arXiv 2606.27098)
   — понижен в приоритете.
