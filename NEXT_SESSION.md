# Continuation prompt — rwkv-quant

Квантование RWKV-7 под Apple Silicon (M4 base, 16GB, БЕЗВЕНТИЛЯТОРНЫЙ Air).
Цель: максимальная производительность decode при минимальной потере качества.
Формат `.rwkvq`, бэкенд Metal (MLX), референс-модель
`~/Develop/WKV-kvant/rwkv7-g1h-1.5b-ctx10240.pth` (1.5B, world naming, ~2.8GB).

## Законы методологии (нарушались, приводило к ложным выводам)

1. Все замеры скорости — ТОЛЬКО A/B-чередованием в одном процессе
   (безвентиляторный дрейф 15.7→26 мс под нагрузкой неустраним).
2. Тяжёлые ppl-прогоны 1.5B — по одному конфигу на процесс
   (`tests/diagnose_one.py <case>`; dense-фолбэк держит fp32, "почти всё
   dense" конфиги прожорливы; однажды 13GB RSS).
3. ppl сравнивать только внутри текущего корпуса
   (eval_corpus_world.pt[:8], построен из ~/Develop/test.txt —
   литература+техника+википедия). Числа других эпох — перемерять.
4. В циклах с деквантом — eval каждую итерацию.
5. Комбинации конфигов непредсказуемы: изолированный выигрыш группы
   не переносится в композит без замера (многократно подтверждено).
6. Metal System Trace GUI: иерархия "Frame" в Encoder Hierarchy НЕ равна
   реальным command buffer'ам (коэффициент расхождения был кратный) --
   для точных чисел экспортировать через `xcrun xctrace export --xpath
   '...table[@schema="metal-application-command-buffer-submissions"]'`
   и парсить САМОСТОЯТЕЛЬНО с учётом id/ref-интернинга экспортного XML
   (повторяющиеся значения выносятся в ref="N", наивный regex по id="N"
   ловит только первую строку). Наивная сумма Duration по всем buffer'ам
   тоже врёт при пересечениях -- нужен union непересекающихся интервалов,
   не sum(). См. сессию 19.07-3 -- на этом получили ложный вывод "72%
   простоя GPU", который сам же и опроверг более аккуратный замер (99%
   занятость). Рецепт headless-запуска: tests/profile_target_champion.py
   в фоне -> ждать STEADY в логе -> xctrace record --attach $PID
   --time-limit Ns -> export -> /tmp/analyze_cmdbuf.py (пока в /tmp этой
   машины, не в репо -- перенести в tests/ при следующем использовании).

## Статус кода (провалидировано)

- Формат `.rwkvq` + INT4 bit-packing: COMPRESSION 1181MB.
- Decode: 15.2 мс/ток (65.8 tok/s) — паритет Q4_K_M llama.cpp (~68).
  Пол по руфлайну: ~9.3 мс/ток (веса/ток / 120 GB/s DRAM M4);
  разрыв = ~6 мс не-GEMV (WKV, 6 lerp token-shift, LoRA fp16, launch).
- GEMV v2-кернель: 90–96 GB/s = 75–80% пика DRAM, почти оптимален;
  v3 (simdgroup) паритет — охота за полосой GEMV малоперспективна.
- GEMM-префилл: 3.6x на T=1024 (1.8 мс/ток; чистые матмулы ~0.7 —
  добирать фьюзом и WKV).
- AW (imatrix-стиль, E[x^2]): собран `/tmp/act_stats_1p5b.pt` (145 ключей,
  сборка ~28с через collect_act_stats.py, устойчив к выбору корпуса).
- writer.py: `_groupwise_fake_dequant(w, bits, gs, sb, sb_bits, ex2)` —
  блочный асимметричный RTN; sb>0 = Q4_K-суперблок (6-бит scale/min против
  fp16-пары на суперблок); sb_bits<0 = грид(0.7..1.05)+LS-поиск scale/min;
  ex2 = AW-взвешенный критерий. `_mxfp4_fake_dequant` (E8M0+E2M1, MSE-грид
  экспоненты, опц. SpQR) — оставлен для истории, в формат не идёт.
  Режимы через `group_scale_mode`: asym / asym_sb6 / asym_sb6_search /
  asym_sb6_aw / mxfp4. ВАЖНО: gw-ветка в quantize_tensor стоит РАНЬШЕ
  act_stats-ветки (иначе per-row-AW перехватывает).

## РЕШЕНИЕ: формат v2 (18.07; АМЕНДМЕНТ 18.07-вечер: смешанная битность)

Раскладка sb6 едина: асимметричный блок 32, суперблок 256, scale/min по
6 бит (qm со сдвигом +31) против fp16-пары d/dm; коды — блок-локальный
split-ниббл (блок-32 = один uint4-лоад), для int5 — битплоскость qh
(1 бит/эл, little-endian по колонкам). bpw: int4 = 4.5, int5 = 5.5.
Значения — AW-взвешенный грид+LS-поиск; d/dm проходят half-роундтрип ДО
выбора qs/qm (кернель = writer бит-в-бит). SpQR из decode-рантайма ушёл.
LoRA w/a/v — gw64@6 asym: int8-контейнер кодов + fp32-пары (крошечные,
жать нечего); g_lora/small — старый per-row int8 (v1).

РЕАЛИЗОВАНО И ПРОВАЛИДИРОВАНО 18.07-вечер (фазы A+B):

## Сессия 19.07-3: compute-ceiling vs bandwidth vs command-buffer-idle

Вопрос сессии: стоит ли целиться в 12 мс/ток, и не упирается ли decode
в ALU/TFLOPS, а не в bandwidth (гипотеза А.). Разобрано по частям,
с поправками по ходу (см. закон №6 выше -- часть выводов сессии сама
себя опровергла на более аккуратном замере).

ФАЗА 1 -- compute ceiling (tests/bench_compute_ceiling.py, эмпирика,
не паспортный TFLOPS): большой compute-bound fp16 матмул на этой M4 =
2733 GFLOPS потолок. Реальные формы decode (proj/cmix/head GEMV, LoRA
batched x24 слоя) держат 70-220 GFLOPS = 2.5-8% потолка. Roofline
точка баланса потолок/bandwidth = 2733/120 = 22.7 FLOP/байт; у decode
арифметическая интенсивность ~1 (dense fp16) -- ~3-4 (int4/int5)
FLOP/байт, на порядок ниже. Вывод держится: GEMV-путь bandwidth-bound
везде, ALU не лимитер, "менять математику" ради FLOP бессмысленно.
GFLOPS реальных форм ~= их же GB/s (fp16: 2 байта/эл, 2 FLOP/эл ->
FLOP:байт~1) -- независимое подтверждение через две разные единицы.

ФАЗА 2 -- живой Metal System Trace (Instruments GUI, GPU Counters):
ALU Utilization 35%, MMU 13%, GPU Read/Bandwidth треки визуально почти
насыщены. 35% ALU НЕ противоречит bandwidth-bound: деквант-математика
(unpack ниббл + sb6 FMA) реальна и не была учтена в FLOP-модели фазы 1
(там мерился чистый dense fp16 GEMV, не квантованный кернель), но она
перекрывается (overlap) с ожиданием памяти -- это нормально для
хорошо сконструированного memory-bound кернела, не признак compute-bound.
Слепая зона: не сверили ALU% с численным bandwidth-% в тот же момент --
не критично, т.к. фаза 3 закрыла вопрос с другой стороны точнее.

ФАЗА 3 -- ЛОЖНЫЙ ВЫВОД И ЕГО ОПРОВЕРЖЕНИЕ (командная граница). По GUI
Encoder Hierarchy ("Frame" в иерархии Process/Frame/CommandBuffer/
Encoder) насчитано 28617 записей / 3.20с занятости на ~11.3с окна ->
ошибочный вывод "72% простоя GPU, командная граница -- главный
пожиратель". ОПРОВЕРГНУТО headless-замером через прямой экспорт схемы
metal-application-command-buffer-submissions (см. закон №6): реальных

## Сессия 19.07-5: REDUCTION v2 -- качество, не скорость

Смена фокуса (по запросу А.): REDUCTION должен быть больше COMPRESSION,
меньше bf16, качество близко к bf16 (не к COMPRESSION) -- база для
QAT/пред-обучения, где даже небольшая деградация заметна при обучении.
Аналогия с MXFP4/MXFP8 (одна и та же схема блочного scale, разная
битность мантиссы) -- REDUCTION должен быть ТОЙ ЖЕ схемой (sb6), просто
с бо́льшим числом бит на код, а не другой техникой ради лучшего числа.

НАХОДКА №1 (важно, меняет план): текущий шипованный REDUCTION-пресет
(presets.py: proj8/cmix8/emb8, w/a/v_lora=4, g_lora=8, small=8),
заявленный в докстринге как "near-lossless, ~2x compression", на деле
даёт ppl=12.8147 (+12.1%) -- НЕ near-lossless. Докстринг неверен,
пресет никогда не был как следует перемерен после фиксов LoRA-комбинаторики
(та же болезнь, что чинили для COMPRESSION -- g_lora=8 уже есть, но
w/a/v_lora=4 всё ещё возможно комбинаторно вредят). Требует отдельного
фикса, независимо от v2.

НАХОДКА №2: "uniform 6-бит, plain asym gs=64" (11.439 ppl, +0.07%,
~1190MB, упоминался в старой таблице как эталон) -- ЛОЖНЫЙ ориентир для
деплоя. У group_scale_mode="asym" НЕТ реального GEMV-кернеля:
quant_model._linear для gw_mode="asym" молча деквантует тензор в ПОЛНЫЙ
dense fp16 при загрузке модели (_dequantize_one -> _DenseLinear) --
комментарий в коде прямо это признаёт ("если попали сюда -- деквант в
fp16-dense, чтобы не падать"). Файл на диске компактный, но в GPU-памяти
и по трафику декода это обычная bf16/fp16-модель без всякого сжатия.
Число 11.439 честное для ppl, но нерелевантно для реального REDUCTION v2
-- под ним нет деплоящегося формата. (Единственное текущее применение
"asym" -- LoRA-ветки, где это не важно: тензоры крошечные, dense-fallback
дёшев.)

НАХОДКА №3 (главная, продуктивная): диагностика proj@6 внутри sb6-схемы
по стадиям конвейера (аналог "Изоляция cmix" 18.07), держа cmix/emb_head
на уровне чемпиона:

| вариант proj-схемы (bits=6) | ppl | vs чемпион (11.710) |
|---|---|---|
| plain sb6 (без поиска, без AW) | 11.6907 | -0.019 |
| sb6 + грид-поиск (без AW) | 11.6948 | -0.015 |
| sb6 + AW (grid+LS+AW) | 11.7298 | +0.020 (ХУЖЕ чемпиона) |
| plain asym gs=64 (не деплоится, см. находку №2) | 11.6493 | -0.061 |

AW-взвешивание (дало -0.68 ppl выигрыша на 4-битных группах в сессии
18.07) на 6 битах ДЛЯ PROJ вредит -- переворачивает знак. Это НЕ общее
правило по битности: для emb_head AW на 6 битах по-прежнему помогает
(p5e6c4 с AW = 11.661 против diag6_emb_sb6plain без AW = 11.775 --
без AW заметно ХУЖЕ). Для cmix@5 без AW результат нейтрален относительно
чемпиона (11.7109 vs 11.7100, изолированно) -- эффект AW/битности
group-specific, закон №5 (комбинации непредсказуемы) в силе И для этого.

ИТОГОВЫЙ КАНДИДАТ REDUCTION v2 (собран по факту изоляции, sb6-схема,
БЕЗ смены формата): proj=6 БЕЗ AW (asym_sb6, plain), emb_head=6 С AW
(asym_sb6_aw, как было), cmix=4 С AW (как чемпион) --

    "reduction_v2_noaw_p6e6c4": ppl=11.6478 (+1.91% vs bf16 11.430)
    ~1054MB (расчёт по формуле bpw: proj 402M el * 6.5/8 + cmix 805M el
    * 4.5/8 + emb_head 268M el * 6.5/8 + LoRA ~57MB; формула bpw=bits+
    фикс.overhead подтверждена на чемпионе: 970.7MB сошлось точно)

Это ЛУЧШЕ "нерабочего" reference'а asym-gs64 (11.6493) при том же духе
схемы, что и COMPRESSION -- бигger sibling, не другая техника. Позиция

ОГРАНИЧЕНИЕ (важно для следующего шага): всё выше -- fake-путь

разрыв ~150MB) И УЖЕ превышает бюджет 5% (+5.62%). Расширение gs
(меньше служебных байт) даёт грошовую экономию (~25MB) ценой ещё
худшего качества (+6.39%) -- не решает разрыв и не должно
использоваться. Пол текущей нибл-упаковки: ~887-970MB в диапазоне
2.4-5.6% -- это уже близко к границе Парето-фронта, не под-оптимизировано.

Для настоящего 4x нужна принципиально новая техника (не крутилка бит):
суб-ниббл/разреженная упаковка (true INT3/INT2 с реальной экономией
байт, не нибл-контейнер) или mixed sparsity/pruning -- ничего из этого
не реализовано и не оценено в этом проекте, отдельная (большая)
инициатива, если понадобится.

## Сессия 19.07-8: int6-расширение кернеля -- REDUCTION v2 разблокирован

Единственный блокер REDUCTION v2 (proj6-noAW/emb6-AW/cmix6-AW, см.
19.07-6) -- реальный packer и Metal-кернель поддерживали только
bits in (4,5). Закрыто по слоям, с гейтом бит-точности на каждом шаге
(как обычно: сначала формат, потом кернель).

**Формат** (schema.py/writer.py/reader.py):
- `QuantizedTensor.gw_qh2` -- вторая битплоскость (6-й бит кода),
  тем же механизмом, что и существующий `gw_qh` (pack_bitplane),
  независимая от него.
- `_make_qt_gw_sb6`: `assert bits in (4,5)` -> `(4,5,6)`; при bits>=6
  извлекается и qh2; добавлен параметр `search: bool`, чтобы реальный
  packer точно повторял поведение fake-пути по режиму (`asym_sb6` --
  фиксированная шкала без перебора, `asym_sb6_search`/`_aw` -- grid+LS
  search) -- иначе фейк/реал разошлись бы по бит-точности молча.
- Попутно найден и закрыт баг: `real_gw=True` диспатч раньше обслуживал
  ТОЛЬКО `mode=="asym_sb6_aw"` -- `"asym_sb6"` (без AW, как раз proj
  REDUCTION v2) упал бы `NotImplementedError` вне зависимости от битности.
- `_dequantize_gw_sb6`: `+ unpack_bitplane(gw_qh2)*32.0` в дополнение к
  существующему `*16.0` для qh.
- Гейт: синтетический тензор 64x256, все 3 режима group_scale_mode при
  bits=6, fake vs real dequant -- `maxdiff=0.000000e+00` во всех трёх.
  Полный регресс (test_v2_format/test_v2_negative/test_gw_kernel) --
  числа идентичны бейзлайну, bits 4/5 не задеты.

**Кернель** (quant_linear_gw.py): `has_qh: bool` обобщён в
`xbits: int` (0=int4, 1=int5, 2=int6) по всему файлу (кеш-ключ,
имя кернеля `quant_linear_gw{4+xbits}_...`, `GwQuantLinear`/
`GwQuantLinearFused`). В шейдере добавлен `qh2_body` -- зеркало
существующего `qh_body`, читает второй uint32/блок из `qh2` и ORит
бит в позицию 5 (вместо 4) нибл-значений l0..h3; включается только при
xbits>=2. `_dequant_w` (GEMM-путь) получил симметричный `*32.0`-вклад
qh2. `GwQuantLinearFused` конкатенирует qh2 по K суб-линейным слоям,
ассерт `xbits` вместо `has_qh` (атрибут `has_qh` оставлен для обратной
совместимости старых бенчей). `input_names` кернеля теперь всегда
включает `qh2` (нулевой placeholder при xbits<2).

Гейт: `test_gw_kernel_int6.py` -- REDUCTION v2 (реальные тензоры
proj/cmix/emb_head, все три group_scale_mode при bits=6), GEMV (N=1,3)
